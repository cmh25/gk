# gk v3 release notes

## additional support for Windows

gk v3 builds and runs natively on Windows three ways:

- **MSVC** — `make.bat` (`cl`, or `clang-cl` via `make.bat gkc`), producing an
  x64 native `gk.exe`.
- **MinGW-w64** — the MSYS2 **UCRT64** environment (`make`), a standalone native
  `.exe` with no emulation runtime.
- **Cygwin** — POSIX at the source level, PE-linked against `cygwin1.dll`
  (`make`).

(WSL has always worked — it's ordinary Linux.) The full test suite passes on all
of them.

### no more carriage returns

gk now prints `\n`, never `\r\n`. On Windows, `stdout` and `stderr` are put in
binary mode (`_setmode(..., _O_BINARY)`), so the C runtime no longer rewrites
line endings on the way out.

The payoff: **gk's output is byte-for-byte identical on every platform.** Binary
data written to `stdout` (e.g. via `` `0: ``) is no longer corrupted by `\n`→`\r\n`
expansion, and redirected output (`gk prog.k > out`) is a clean LF file. gk is no
longer the source of stray carriage returns in your pipelines.

The Windows console renders a bare `\n` as a proper newline, so interactive use
and the REPL are unaffected. `stdin` is left in text mode: incoming `\r\n` is
still normalized to `\n`, and gk's reader was already `\r`-tolerant regardless.

### foreign functions and libm conformance

- `2:` (link object code / FFI) now works on **Cygwin** and **MinGW**. A PE plugin
  DLL can't leave gk's `gk_*` callbacks unresolved the way an ELF `.so` can, so
  gk exports an import library the plugin links against. The shim source is
  unchanged across platforms.
- `gamma` at its negative-integer poles now returns `0n` (NaN) on every platform.
  Some libm implementations (notably Cygwin's newlib) returned `0i` (infinity)
  there; gk now guards the poles itself so the result is deterministic regardless
  of the C library.

## deeper recursion

gk previously capped recursion at a fixed, conservative depth (~239 on an 8MB
stack) so that deep nesting raised a clean `stack error` instead of overflowing
the C stack and crashing. That cap was both too shallow for ordinary recursive
code and a blunt instrument: a single number can't account for how much stack
each level actually uses.

v3 guards on the **actual remaining stack** instead. gk checks how much stack is
left as it recurses and raises a clean `stack error` just before it would run out.
The effect is adaptive — lightweight recursion now runs **thousands of levels deep**
(~2000–4000 on an 8MB stack, roughly 12× the old limit), while a heavy per-level
body stops earlier, exactly when the stack is genuinely about to be exhausted.
Either way you get a clean error, never a segfault.

The stack bounds are queried from the operating system on each platform (Linux,
macOS, Windows, Cygwin), so the limit tracks the real stack and the behavior is
consistent everywhere. This is purely more permissive — code that ran before still
runs; code that used to hit the old shallow cap now keeps going.

## non-upward compatible changes (NUCs)

### 1. command-line arguments (`.z.i`)

gk now collects the command-line arguments into `.z.i` the way k3 (and the
reference interpreter) does: the interpreter name, every flag (`-q`, `-i`, `-f`),
and the script path itself are stripped, leaving only the user arguments. **Flag
position no longer matters.**

```
  .z.i   / invoked as:  gk prog.k -i 5555 -q one two
("one"
 "two")
```

Previously gk assumed the script was always the first argument, so flags (and the
script path) could leak into `.z.i` depending on argument order. The new behavior
is consistent and order-independent.

### 2. Windows output is LF, not CRLF

Covered above. If you have Windows scripts or tooling that depended on gk emitting
`\r\n`, they will now see `\n`. This is the change that makes v3 a major release on
Windows.

### 3. `6:` append is binary on Windows

`6:` (append bytes to a file) now opens the file in binary mode on Windows, so raw
bytes are written verbatim. Previously the file was opened in text mode and any
`\n` byte was expanded to `\r\n`, corrupting binary appends. (On other platforms
`6:` was already binary; this only changes Windows.)
