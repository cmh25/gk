# gk v2 release notes

## new data types

gk v2 adds two new data types: 64-bit integers and 32-bit floats.

### 1. 64-bit integer

A 64-bit signed integer, written with a `j` suffix.

```
  1j
1j
  0 1 2j
0 1 2j
  2j + 3            / i32 promotes to i64
5j
  4:1 2 3j          / type is 8 (atom) / -8 (vector)
-8
  0Nj 0Ij -0Ij      / null, infinity, -infinity
0Nj 0Ij -0Ij
  1000000000000j * 1000000j
1000000000000000000j
  */1+!20j
2432902008176640000j
```

i64 is supported across the whole verb and builtin surface.

### 2. 32-bit float

A 32-bit float, written with an `e` suffix.

```
  1.0e
1.0e
  1 2 3.0e          / any e-suffix makes the whole literal f32
1 2 3.0e
  4:1.0 2.0e        / type is 9 (atom) / -9 (vector)
-9
  0ne 0ie -0ie      / null, infinity, -infinity (lowercase, like f64)
0ne 0ie -0ie
```

An f32 operation stays f32 when every operand is i32 or f32. It only widens to f64 if
an i64 or f64 is involved.

```
  sqrt 2.0e
1.414214e
  6.0e%2
3.0e
  %2.0e
0.5e
  2.0e^10
1024.0e
```

Comparison uses a tolerance scaled to single precision, so single-bit round-off
compares equal:

```
  ((sqrt 2.0e)*sqrt 2.0e)=2.0e
1
```

f32 is supported across the whole verb and builtin surface.

Note on `e` and scientific notation: a digit immediately after `e+`/`e-` is read as a
scientific exponent, so `1.0e+2` is `100.0`. To add an f32 and an i32, separate them
(`3 + 2.0e`), and an f32 literal disambiguates on its own (`1.0e+2.0e` is `3.0e`).

### 3. numeric promotion

The four numeric types promote to the wider type. The only non-obvious rule is i64 + f32:
neither holds the other, so the result widens to f64.

```
i32 + i64 -> i64
i32 + f32 -> f32
i32 + f64 -> f64
i64 + f32 -> f64
i64 + f64 -> f64
f32 + f64 -> f64
```

```
  4:1 + 1j          / i32 + i64 -> i64
8
  4:1.0e + 1        / f32 + i32 -> f32
9
  4:1j + 1.0        / i64 + f64 -> f64
2
  4:1.0e + 1j       / f32 + i64 -> f64
2
  4:1.0e + 1.0      / f32 + f64 -> f64
2
```

## performance

### force monad

In some contexts the only way to force a primitive to its monadic form is with a lambda,
e.g. `{,x}` to force `,` to enlist. In gk v2 such lambdas are implemented efficiently,
with the same performance as the primitive itself.

## non-upward compatible changes (NUCs)

### 1. k data binary format (1:, 2:, 5:)

The serialized format version was bumped to carry the new types. The change is additive,
so **v2 reads data written by v1** — but data written by v2 is **not** readable by v1
(v1 rejects the newer version). Upgrade is safe; downgrade is not.

### 2. assigning to a builtin or predefined name is a reserved error

```
  inv:3
reserved error
inv:3
   ^
>
```

Previously this produced a confusing `type error`.

### 3. floor (_) uses comparison tolerance

`_` now uses comparison tolerance, consistent with `=`, `<`, and `>`: a value
tolerantly equal to an integer floors to that integer.

```
  _ 1.9999999999999        / tolerantly 2
2
  _ 1.999999999999         / not tolerantly 2
1
```

(The builtin function `floor` remains the exact, tolerance-free version.)

### 4. mod (!) with a negative divisor

`!` is now true floored modulo, `x - y*_(x%y)`; the result takes the sign of the
divisor.

```
  5!-3
-1
```

In v1 this returned `2`. Mod by a positive divisor is unchanged.

### 5. fixed-width `1:` float letters

The new `i64` and `f32` types gained fixed-width field codes: text `0:` adds
`J` (i64) and `E` (f32) alongside `I F C S`, and binary `1:` adds `j` (i64)
and `e` (f32). The text additions are purely additive.

The binary float codes were also realigned so lowercase mirrors the uppercase
text codes by width — `i`/`j` = 4-/8-byte integer, `e`/`f` = 4-/8-byte float:

| code | v1 | v2 |
|------|----|----|
| `e` | — | 4-byte → `f32` |
| `f` | **4-byte** float → `float` (f64) | **8-byte** → `float` (f64) |
| `d` | 8-byte double → `float` (f64) | **removed** |

So `f` now reads 8 bytes (was 4) and `d` no longer exists. To port a v1 schema:
an 8-byte double field `d` becomes `f`; a 4-byte float field `f` becomes `e`
(now read natively as f32 — promote to f64 in-language if needed). This applies
to both the column form `(s;w)1:f` and the single-letter whole-file form
(`"f"1:f` reads f64; `"d"1:f` is gone).
